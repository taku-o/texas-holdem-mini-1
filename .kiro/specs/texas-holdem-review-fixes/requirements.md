# Requirements Document: レビュー指摘対応

## Introduction

本仕様は、テキサスホールデム Web アプリ（`texas-holdem-webapp`）に対するレビュー指摘事項を要件として定義する。指摘はドメイン層（betting, dealing, showdown, handProgression, handEvaluator, cpuStrategy, gameSetup）、アプリケーション層（gameFlow, useGameController）、UI層（TableView, PlayerSeat, ActionBar）にわたる。既存の `texas-holdem-webapp` 仕様を前提とし、堅牢性・一貫性・アクセシビリティの改善を目的とする。

## Project Description (Input)

次のレビュー指摘事項に対応したい。

- **HIGH**: ベット/レイズ額のバリデーション不足、ベッティングラウンド終了判定の無限ループリスク
- **MEDIUM**: レイズ可否判定、BB不足時の currentBet、サイドポット未対応、チップ0プレイヤーの次ハンド参加、CPU 同期ブロック
- **LOW**: 脱落プレイヤーへのブラインド、ショートスタック時のレイズ額、ロイヤルフラッシュ判定の依存、getValidActions の amount、乱数結合、key の使用、クライアント側バリデーション、アクセシビリティ、functional setState

---

## Requirements

### 1. ベッティングのバリデーション（betting.ts）

**目的:** ベット/レイズ額の検証を厳格にし、チップが負になる不正状態を防ぐ。

#### 受け入れ基準

1. When 人間または CPU がベットまたはレイズを行う場合、the system shall `action.amount` が当該プレイヤーの所持チップ以下であることを検証し、違反時は状態を更新せずエラーを返す。
2. When ベットまたはレイズを行う場合、the system shall レイズ額が最低レイズ額以上であること、およびルール上有効な範囲であることを検証し、違反時は状態を更新せずエラーを返す。
3. When レイズを選択可能か判定する場合、the system shall 当該プレイヤーが最低レイズ額を支払えるか（所持チップがコール額＋最低レイズ額以上か）をチェックし、支払えない場合はレイズを有効にしない。

---

### 2. ベッティングラウンド終了判定（betting.ts）

**目的:** オールインした last aggressor によりベッティングラウンドが終了しない無限ループを防ぐ。

#### 受け入れ基準

1. When ベッティングラウンドの終了条件を判定する場合、the system shall last aggressor がオールイン（チップ0）でアクティブプレイヤーから除外されている状況を考慮し、当該プレイヤーが次のアクティブプレイヤーとして戻ってこない場合でもラウンドが正しく終了するようにする。
2. The system shall `isBettingRoundComplete`（または同等の終了判定）が、last aggressor がスキップされる場合でも有限回で true となり、無限ループが発生しないようにする。

---

### 3. ブラインド・ショートスタック（dealing.ts）

**目的:** ショートスタック時の currentBet と、脱落プレイヤーへのブラインド課金を正しく扱う。

#### 受け入れ基準

1. When ビッグブラインドをポストするプレイヤーがショートスタック（BB 額未満）の場合、the system shall 実際にポストした額を `currentBet` に設定し、`currentBet = BIG_BLIND` とはしない。
2. When 各ハンドでブラインド（SB/BB）を課す場合、the system shall チップが 0 などで脱落済みのプレイヤーにはブラインドをポストさせず、アクティブなプレイヤーのみから SB/BB を徴収する。

---

### 4. ショーダウン・ポット配分（showdown.ts）

**目的:** オールイン時のポット配分の正当性を定義する（サイドポット対応の有無を明示する）。

#### 受け入れ基準

1. When ショーダウンで複数プレイヤーがオールインしている場合、the system shall 単一ポットの簡略化仕様の範囲で、ポット配分がルール上不正にならないようにする（サイドポットを実装するか、または「単一ポット・オールイン時は簡略化された配分」であることを仕様として明記し、不正なマイナスチップ等が発生しないようにする）。

---

### 5. チップ0プレイヤーと次ハンド（handProgression.ts）

**目的:** チップが 0 になったプレイヤーを次ハンドから除外する。

#### 受け入れ基準

1. When 1 ハンド終了後にチップが 0 のプレイヤーがいる場合、the system shall そのプレイヤーを次のハンドに参加させない（除外またはシットアウトとする）。
2. The system shall 次ハンドの席順・ディーラーボタン・ブラインドの計算において、除外されたプレイヤーをスキップする。

---

### 6. CPU のレイズ額（cpuStrategy.ts）

**目的:** ショートスタック時でも CPU のレイズ額がルール上有効であるようにする。

#### 受け入れ基準

1. When CPU がレイズを選択する場合、the system shall レイズ額が `currentBet`（またはコールに必要な額）以上となるようにし、ショートスタック時に `currentBet` 未満の額が設定されないようにする。

---

### 7. 役判定の依存（handEvaluator.ts）

**目的:** ロイヤルフラッシュ判定がライブラリの内部定数に直接依存しないようにする。

#### 受け入れ基準

1. The system shall 役判定（ロイヤルフラッシュ含む）を、外部ライブラリの内部定数に直接依存しない形で利用する（アダプタでラップする、または必要な定数をアプリ側で定義して比較する等）。

---

### 8. 有効アクションの返却形式（betting.ts）

**目的:** UI がベット/レイズを提示する際に必要な情報を欠かさない。

#### 受け入れ基準

1. The system shall `getValidActions`（または同等の API）がベット/レイズを返す場合、選択可能な額（amount または min/max）を含め、クライアントがそのままチップ入力の範囲として利用できるようにする。

---

### 9. 乱数関数の結合（gameSetup.ts）

**目的:** テストや再現性のため、乱数関数を差し替え可能にする。

#### 受け入れ基準

1. The system shall ゲーム開始時の席決定などで用いる乱数生成を、差し替え可能な形（引数で渡す、または依存注入）で結合し、テストでは固定シードやモックを注入できるようにする。

---

### 10. CPU 処理の非同期化（gameFlow.ts）

**目的:** 大量の CPU ターンでもメインスレッドを長時間ブロックしないようにする。

#### 受け入れ基準

1. The system shall CPU の連続処理（複数ターンや複数プレイヤーの決定）を、同期的にメインスレッドをブロックしすぎない形で実行する（例: 一定件数ごとに yield する、または非同期で次のティックに回す等）。現状のプレイヤー数で問題がなくても、アーキテクチャ上その方針を満たすようにする。

---

### 11. リスト表示の key（TableView / PlayerSeat）

**目的:** React の一意性・再描画の安定性のため、index ではなく安定した ID を使う。

#### 受け入れ基準

1. The system shall テーブル上の席やカードのリストを描画する際、`key` に配列の index ではなく、プレイヤー ID やカード ID など安定した一意の識別子を使用する。

---

### 12. チップ額入力のクライアント側バリデーション（ActionBar）

**目的:** 送信前に不正なチップ額を防ぎ、UX を改善する。

#### 受け入れ基準

1. When 人間プレイヤーがベット/レイズでチップ額を入力する場合、the system shall クライアント側で最小・最大・所持チップの範囲内であることを検証し、範囲外の値では送信を無効化するかエラーメッセージを表示する。

---

### 13. アクセシビリティ（スライダー・入力・カード）

**目的:** スクリーンリーダーやキーボード操作に対応する。

#### 受け入れ基準

1. The system shall チップ額用のスライダーおよび数値入力に、`label` または `aria-label` 等で意味が分かるラベルを付与する。
2. The system shall カードの裏面画像（またはプレースホルダー）に、`alt` テキストまたは同等のアクセシブルな説明を付与する。

---

### 14. ゲーム開始時の setState（useGameController.ts）

**目的:** 初期状態の確実な反映のため、functional setState を使う。

#### 受け入れ基準

1. When ゲーム開始（startGame）で状態を初期化する場合、the system shall 前回の状態に依存しないように、functional setState（`setState(prev => ...)` の形でなく、初期状態を直接設定する、または `setState(() => newState)` のように関数形式で新しい状態を返す）を用いる。
